# GPT-5.4 与 Gemini 产出质量对比分析

## 1. 结论先行

如果只看**当前已经落盘的结果质量**，`gpt-5.4` 的效果整体更好，尤其体现在：

- `facts` 抽取更完整，能把一个 scene 拆成更多有效事件、关系变化和制度性设定。
- `style` 蒸馏更像“可直接拿来写 RP / 世界书 / 风格手册”的机制说明，而不只是主题总结。
- 对这本书最关键的“资源压力 / 债务压力 / 社会羞耻 / 制度荒诞 / 黑色幽默”几个核心轴线，`gpt-5.4` 的命名和归纳更到位。

如果把“能否稳定跑完整库”也算进“效果”，那当前生产环境下仍然是 `Gemini` 更合适扛第一轮全库蒸馏，因为它已经在持续运行，而 `gpt-5.4` 只在小规模实验里稳定。

一句话判断：

- **首轮全库蒸馏**：继续用 `Gemini`
- **关键章节复核 / 风格总手册 / 高价值设定精修**：优先用 `gpt-5.4`

## 4. 质量层面的核心差异

## 4.1 `gpt-5.4` 的优势

### A. 更会拆 scene 的内部节拍

典型例子：`scene_0001_001.json`

- `gpt-5.4` 拆出了 6 个事件、2 个关系变化。
- `Gemini` 只保留了 1 个总括性事件，没有关系变化。

这意味着：

- `gpt-5.4` 更适合后续做剧情节点、角色关系演化、情节回放。
- `Gemini` 更像是在做“压缩摘要”，而不是“可追踪 scene 结构”。

### B. 更能抓住这本书真正重要的“制度性信息”

这本书最难抽的不是人名地名，而是：

- 贫穷如何驱动行为
- 债务如何塑造身份
- 学校/宗门/金融/医疗如何联动
- 黑色幽默如何来自“制度口吻 + 极端内容”的错位

在这些点上，`gpt-5.4` 明显更强。

例如在 `style_window_0001_0002.json` 中，`gpt-5.4` 能直接说出：

- “羞耻型喜剧反转”
- “反向考古式塑造”
- “个人窒息感 + 系统性压迫感”的双重视角
- “向上流动必须先付费”的叙事结构

这些描述已经接近“风格手册条目”，而不是普通文学赏析。

### C. 更像“二次加工后的可用知识”

同样是 `facts`，`gpt-5.4` 的结果更方便后续直接进入：

- 主线节点整理
- RAG 事实库
- 世界书条目归类
- 风格规则提炼

原因是它更愿意把：

- 事件拆开
- 关系变化显式写出
- scene 内可见的风格机制单独标记

## 4.2 Gemini 的优势

### A. 更克制、更短、更适合第一轮全量铺开

`Gemini` 的结果平均更短，字段更少，虽然信息密度不足，但也带来两个现实好处：

- 单条结果更容易快速审读
- 对首轮全库跑批更友好

### B. 有时会抓到一些 `gpt-5.4` 漏掉的显式细节

例如 `scene_0002_004.json`：

- `Gemini` 明确抽出了“白真真欠债二十多万”
- 同 scene 的 `gpt-5.4` 没把这条单独落成 fact

这说明 Gemini 并不是“全面更弱”，而是：

- 它的**局部显式信息命中**有时不错
- 但它整体上更少做结构展开

### C. 当前生产可用性更高

从当前工程状态看：

- `Gemini` 正在全库正式运行
- `gpt-5.4` 目前仍停留在小样本稳定实验

所以“生产效果”不能只看单条质量，还要看能不能跑完。

## 4.3 Gemini 的主要短板

### A. 把多段情节压成一个事件

最常见问题是：

- 一个 scene 明明有“进入场景 -> 认知冲击 -> 制度揭露 -> 关系变化”
- `Gemini` 往往只保留 1-3 个大事件

这会直接损失后续：

- 剧情节点切分精度
- 角色弧线追踪能力
- “一个 scene 里哪一拍最关键”的可见性

### B. 风格结论偏“主题词”，不够“机制词”

`Gemini` style 的典型表述是：

- 社会批判
- 黑色幽默
- 债务压力
- 分数至上

这些并没有错，但它经常停在“主题识别”，没有继续往下拆成：

- 这个效果是怎么生成的
- 作者具体用什么叙事动作实现
- 哪类句法、哪类转折、哪类身份反差在反复出现

### C. 更容易出现 schema 漂移或不够严谨的推断

几个典型问题：

- 空 `object` 的 fact 更多
- `relationship_changes` 经常缺失
- 某些 scene 会给出略显过强的解释性推断

例如 `scene_0002_001.json` 中，Gemini 给出“张羽是穿越者或夺舍者”的 inferred fact。

站在全书知识上这件事大概率没错，但如果只按单个 scene 的取证强度看，这条推断偏猛。

## 5. 样本级例子

## 5.1 `scene_0001_001.json`

`gpt-5.4` 更好。

原因：

- 把“母亲短信安慰”“进入面试”“成绩确认”“睡眠差距”“课程差距”“绝育要求”拆成了 6 个事件。
- 明确写出“母亲 -> 张羽”的支持关系强化，以及“张羽 -> 嵩阳高中”的认知变化。
- `power_system_notes` 里把“绝育要求”“睡眠标准”“课程起点”“激素管理”拆开写，适合进设定库。

Gemini 的问题：

- 几乎把全部内容压成 1 个总事件。
- `relationship_changes` 空缺。
- 把 `嵩阳高中` 和 `东阳初级中学` 都打成 `location`，不如 `gpt-5.4` 的区分更贴合“学校/势力机构”语义。

## 5.2 `scene_0002_001.json`

`gpt-5.4` 更稳。

原因：

- 它把“考试噩梦”“惊醒”“判断梦与原身记忆有关”“记忆未融合”“回忆仪式受阻”“五点自然醒”拆得很清楚。
- 不急着下“穿越者/夺舍者”结论，而是先停在“掌握这具身体但记忆未融合”。

Gemini 的问题：

- 结构太薄，只保留了 2 个事件。
- 给出“张羽是穿越者或夺舍者”的 inferred fact，证据链略偏强推。

## 5.3 `scene_0002_004.json`

这一组更接近“互有优劣”。

`gpt-5.4` 优势：

- 更系统地拆出“查看花费 -> 风险说明 -> 规则回忆 -> 劝退 -> 转账 -> 迷茫表态”。
- 关系变化写得更完整，尤其是“怜悯 -> 劝退 -> 转账 -> 感谢”这条线。
- `power_system_notes` 的组织更适合后续世界设定索引。

Gemini 优势：

- 明确抽出了“白真真欠债二十多万”这条显式信息。

结论：

- 如果目标是“事实颗粒度 + 关系变化 + 后续节点整理”，还是 `gpt-5.4` 更好。
- 如果目标只是“快速知道本 scene 在讲什么”，Gemini 也够用。

## 6. 风格蒸馏哪边更好

结论非常明确：**`gpt-5.4` 更好，而且差距比 `facts` 更明显。**

原因不是它“更会夸”，而是它更会把风格写成可操作机制：

- 它会解释“为什么好笑”
- 它会解释“为什么荒诞”
- 它会解释“人物为什么这样立得住”
- 它会解释“这部作品为什么不是标准修仙文”

而 Gemini 更多是在做：

- 题材归纳
- 主题标签
- 大方向概括

对于你真正要做的事情：

- RP 设定
- 世界书
- 风格模仿
- 主线判定

显然 `gpt-5.4` 这种“机制级输出”更有价值。

## 7. 最终判断

### 7.1 只看质量

`gpt-5.4` 胜出。

而且不是微弱领先，是在下面几个点上都更明显：

- scene 拆解能力
- 关系变化提取
- 制度信息命名
- 风格机制提炼
- 对本书“非标准修仙”本质的把握

### 7.2 看当前工程现实

`Gemini` 仍然是当前更合适的首轮全库方案。

因为：

- 它已经在跑
- 它能稳定续跑
- 它的结果虽然偏简，但已经能形成 usable 的第一层结构数据

### 7.3 推荐落地策略

最适合你的不是二选一，而是分层：

1. `Gemini` 负责全库首轮 `facts`
2. `Gemini` 先出一版全库 `style/canon/review`
3. `gpt-5.4` 专门接管以下高价值任务：
   - 前 50 章风格总手册
   - 关键人物与关键势力设定复核
   - 主线剧情节点整理
   - 高价值章节的二次蒸馏
   - 世界书条目的语言润色与机制归纳

# 文风蒸馏器.md 适配分析

## 8. 结论先行

`文风蒸馏器.md` **有用，但不能直接照搬进当前流水线**。

更准确地说：

- 它很适合做“二阶段风格总编排器”
- 不适合直接替代我们当前的 `style_extraction.md`

## 9. 它有价值的部分

### A. 三层结构很强

它把风格拆成：

- `narrative_system`
- `expression_system`
- `aesthetics_system`

这是非常有价值的，因为它逼模型不要只说“这本书很黑色幽默”，而要继续拆：

- 叙述距离怎么控制
- 节奏怎么推进
- 对话怎么生效
- 人物如何刻画
- 审美目标是什么

这部分非常适合拿来改造我们后续的 `style bible`。

### B. 它自带质检思路

`phase 3` 的质检理念很有价值：

- 术语准确
- 参数与例证互相验证
- 可执行
- 有正向指令也有负向约束

这比单纯要模型“总结风格”强很多。

### C. 它天然适合做“风格生成规则”

如果目标是：

- 让模型学这种文风去写 RP
- 输出可供世界书/提示词使用的风格规则

那么它里面这种“结构化风格配置”的思路是对的。

## 10. 它不适合直接照搬的部分

### A. 它是“作者研究器”，不是“小说蒸馏器”

它的输入范式是：

- 作家名
- 代表作
- 搜索结果
- 文学评论
- 学术术语

而我们现在做的是：

- 对单部小说全库做工程化蒸馏
- 服务 RP / RAG / 世界书 / 节点路由

这两者不是同一任务。

### B. 它过度依赖外部搜索与学术引用

这对我们当前目标并不友好，原因有三：

1. 我们已经有整本小说本地语料，不缺文本样本
2. 外部搜索会引入不必要的噪音和成本
3. 学术引用不是当前产物的刚需，反而会占掉 token 预算

### C. XML + 固定 token 目标不适合现在的工程链路

当前流水线更适合：

- JSON
- 稳定 schema
- 易进 RAG / 审核面板 / review panel / canon builder

而不是 XML artifact。

### D. 它对版权与 payload 都比较重

它要求：

- 多段原文摘引
- 每段 30-80 字
- 多个 example

对我们这种全库批处理来说：

- token 很贵
- payload 很重
- 还会放大引文管理成本

### E. 它没有覆盖我们最重要的工程维度

当前项目最重要的并不只是“文学风格”，而是：

- 债务压力如何驱动剧情
- 制度如何塑造角色选择
- 羞耻感如何塑造人物关系
- 主线节点如何切
- 哪些设定进 RAG，哪些进世界书

`文风蒸馏器.md` 没有直接为这些任务设计字段。

## 11. 最适合的使用方式

建议把它改造成**二阶段风格总手册 prompt**，而不是直接替换当前 prompt。

### 11.1 建议保留

- 三系统结构：叙事 / 表达 / 美学
- 质检阶段
- “例证必须支撑结论”的原则
- 对可执行风格规则的强调

### 11.2 建议删除

- 外部搜索 phase
- 学术引用硬要求
- XML 输出
- 作家级泛化框架
- 固定 token 目标

### 11.3 建议新增

针对本项目，新增这些字段会更实用：

- `voice_contract`
  - 叙述口吻、吐槽强度、冷感程度、是否保留现实化术语
- `pressure_axes`
  - 资源压力 / 债务压力 / 阶层压力 / 身体异化 / 羞耻管理
- `humor_recipe`
  - 制度口吻 + 极端内容 + 主角吐槽 + 身份反差
- `character_arc_rules`
  - 人物如何通过贫穷、劳动、债务、排名、体面崩塌来塑造
- `dialogue_rules`
  - 对话长度、功能密度、是否带“产品说明/规章制度/贷款话术”口吻
- `worldbook_binding`
  - 哪类内容适合进入世界书，哪类只做风格规则
- `negative_rules`
  - 不要把文本写成传统热血修仙爽文
  - 不要把阶层压力写没
  - 不要把身体异化写成单纯猎奇

## 12. 最终建议

对 `文风蒸馏器.md` 的最终判断：

- **不是没用**
- **而是不能直接拿来跑现在这条流水线**
- **最适合改造成“全库蒸馏完成后的风格总编排器”**

推荐落地顺序：

1. 先让当前 `Gemini` 跑完整库第一轮
2. 从全库 `facts/style/canon` 中挑出高价值章节窗口
3. 用 `gpt-5.4` 做二阶段风格总手册
4. 参考 `文风蒸馏器.md` 重写一版“项目专用风格综合 prompt”

## 13. 一句话结论

- `gpt-5.4` 的 experimental 结果，质量上整体优于当前可见 Gemini 结果。
- `Gemini` 目前的最大优势不是“更聪明”，而是“更适合先把整库跑完”。
- `文风蒸馏器.md` 值得吸收方法论，但不建议原样接入；应改造成我们的二阶段风格总手册生成器。
